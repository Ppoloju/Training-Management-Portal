A full-stack guide for new developers — from architecture to daily workflows.

1. Introduction

What the Project Does

The Training Management Portal is a corporate web application that helps organizations manage employee training. Administrators can maintain employee and trainer records, define courses and training programs, schedule sessions, enroll employees, and track attendance.

Purpose of the Application

It centralizes training operations in one place instead of spreadsheets or email. The goal is to streamline professional development: who is trained, what programs exist, who attends sessions, and whether they were present.

The frontend calls the backend over HTTP. The backend validates requests, applies business rules, and reads/writes data through Spring Data JDBC repositories. JWT tokens secure all API calls except login.

3. Backend Explanation

What is Spring Boot?

Concept: Spring Boot is a Java framework that builds production-ready web applications quickly. It auto-configures servers, databases, and security so you focus on business code.
In this project: TrainingManagementPortalApplication.java is the entry point. It starts the embedded Tomcat server and loads all Spring components.

What is a REST API?

Concept: A REST API exposes data and actions over HTTP using URLs and methods (GET, POST, PATCH, DELETE). Clients send JSON and receive JSON responses.
In this project: All APIs live under /api/v1/ (e.g. GET /api/v1/employees).

What is a Controller?

Concept: A Controller is the front door of the API. It maps URLs to methods and returns HTTP responses.
In this project: EmployeeController, CourseController, AuthController, and others. Each uses @RestController and @RequestMapping.

What is a Service?

Concept: Services hold business logic and database transactions. Controllers should not query the database directly.
In this project: EmployeeServiceImpl handles save, find, and soft-delete. Interfaces like EmployeeService define the contract.

What is a Repository?

Concept: Repositories are the data-access layer. You define an interface; Spring generates query implementations.
In this project: EmployeeRepository extends CrudRepository with methods like findByIdAndDeletedFalse.

What is an Entity?

Concept: An Entity (model) is a Java class that represents a database table row.
In this project: Employee, Course, Enrollment in the model package use @Table from Spring Data JDBC.

What is a DTO?

Concept: A Data Transfer Object carries data between client and server. It exposes only the fields the API should show or accept.
In this project: EmployeeCreateRequest (input) and EmployeeResponse (output) in dto/request and dto/response.

What is application.properties?

Concept: Configuration file for database URLs, JWT settings, logging, and profile-specific overrides.
In this project: application.properties sets JWT secret, Liquibase path, and default profile local. application-local.properties sets PostgreSQL connection details.

Dependency Injection in This Project

Concept: Spring creates objects and injects them where needed. You declare dependencies via constructor parameters instead of calling new.
In this project: @RequiredArgsConstructor (Lombok) + private final EmployeeHandler employeeHandler in controllers. Spring wires Handler → Service → Repository automatically.

Note: This project also uses a Handler layer between Controller and Service. Handlers orchestrate validation, mapping, and enrichment before calling services.

7. Repositories

Spring Data (JDBC in This Project)

This project uses Spring Data JDBC (not JPA). The repository pattern is the same: interfaces extend CrudRepository and Spring generates implementations. There are no JPA relationship annotations; foreign keys are stored as UUID columns.

Repositories handle SQL operations without writing JDBC code manually. Custom query methods follow naming conventions (e.g. findAllByDeletedFalse).

8. Entities

All entities use UUID primary keys, audit fields (createdAt, updatedAt, createdBy, updatedBy), and a deleted flag for soft deletes. Relationships are stored as UUID foreign key columns (not JPA @ManyToOne).

Employee

Table: employee
Fields: firstName, lastName, email, department, isActive
Relationships: Referenced by Enrollment via employee_id.

Trainer

Table: trainer
Fields: firstName, lastName, email, specialization, isActive
Relationships: Referenced by TrainingProgram via trainer_id.

Course

Table: course
Fields: title, description, startDate, endDate, status (DRAFT, PUBLISHED, ARCHIVED)
Relationships: Referenced by TrainingProgram via course_id.

TrainingProgram

Table: training_program
Fields: title, description, startDate, endDate, status, maxParticipants, trainerId, courseId
Relationships: Many programs → one Trainer (trainerId); many programs → one Course (courseId). One program → many Sessions.

TrainingSession

Table: training_session
Fields: title, description, startDate, endDate, location, capacity, trainingProgramId
Relationships: Many sessions → one TrainingProgram. Referenced by Enrollment.

Enrollment

Table: enrollment
Fields: employeeId, trainingSessionId, enrolledAt
Relationships: Links Employee to TrainingSession. Referenced by Attendance.

Attendance

Table: attendance
Fields: enrollmentId, attendedAt, present
Relationships: Many attendance records → one Enrollment.

9. DTOs

Why DTOs Are Used

  • Hide internal fields (e.g. deleted, newEntity) from API clients.
  • Validate input separately from database models.
  • Return enriched data (e.g. EnrollmentResponse includes employee name).

DTO vs Entity

EntityDTO
Maps to database tableMaps to API request/response
Has audit and soft-delete fieldsExposes only business fields
Used inside services/repositoriesUsed at API boundary

Request DTOs (examples)

EmployeeCreateRequest, CourseCreateRequest, EnrollmentCreateRequest, AuthRequest — sent from client to server.

Response DTOs (examples)

EmployeeResponse, CourseResponse, EnrollmentResponse, PageResponse, AuthResponse — returned to client.

MapStruct mappers in mapper/ convert between DTOs and entities.

10. Frontend Explanation

React Project Structure

The frontend is a Single Page Application (SPA) built with Vite. Entry point: main.tsx → App.tsx. Source lives in frontend/src/.

Components

Reusable UI pieces in components/: DashboardCard, PageHeader. Pages compose these with MUI components (Tables, Dialogs, Buttons).

Pages

Full screens in pages/: Dashboard, Employees, Trainers, Courses, Training Programs, Sessions, Enrollments, Attendance, Assignments, Login, Profile, error pages.

Routing

React Router in App.tsx. Public route: /login. Protected routes wrap Layout via PrivateRoute, which redirects unauthenticated users to login.

API Service Layer

services/api.ts creates an Axios instance with base URL http://localhost:8080/api/v1. It attaches JWT from localStorage and redirects on 401. Feature services (employeeService.ts, etc.) call this client.

State Management

  • AuthContext — token, login/logout, isAuthenticated.
  • TanStack React Query — server data fetching, caching, mutations (used on list pages).
  • Local useState — form fields, dialogs, pagination on individual pages.

Forms

Login uses React Hook Form + Yup validation. CRUD pages (e.g. Employees) use MUI TextField with local state and submit via React Query mutations.

API Calls

Pages call service modules → Axios → Spring Boot. Example: EmployeesPage uses useQuery with employeeService.getEmployees().

14. Complete Project Workflow

End-to-end: enrolling an employee in a training session.

1. Open browser → http://localhost:4173
2. LoginPage → POST /api/v1/auth/login
3. JWT stored in localStorage
4. Navigate to Enrollments or Assignments page
5. Select employee + training session
6. enrollmentService → POST /api/v1/enrollments
7. EnrollmentController → EnrollmentHandler
8. EnrollmentServiceImpl → EnrollmentRepository
9. Row inserted in enrollment table (PostgreSQL)
10. EnrollmentResponse returned → UI table updates

16. How This Project Works

  1. The app manages corporate training: employees, trainers, courses, programs, sessions, enrollments, and attendance.
  2. React frontend (Vite + MUI) provides the UI; Spring Boot backend exposes REST APIs.
  3. PostgreSQL stores all data; Liquibase applies schema migrations on startup.
  4. Users log in via /login; the backend returns a JWT stored in localStorage.
  5. Every API call (except login) sends the JWT in the Authorization header.
  6. Controllers receive HTTP requests and delegate to Handlers.
  7. Handlers validate input, map DTOs to entities, and call Services.
  8. Services run business transactions and call Repositories.
  9. Repositories use Spring Data JDBC to read/write PostgreSQL tables.
  10. Entities map to tables; foreign keys are UUID columns (trainer_id, course_id, etc.).
  11. DTOs separate API contracts from database models; MapStruct converts between them.
  12. Soft deletes set deleted=true instead of removing rows.
  13. Frontend pages use React Query to fetch and cache API data.
  14. Service modules in frontend/services/ mirror backend endpoints.
  15. Training flow: Course + Trainer → Program → Session → Enrollment (Employee) → Attendance.
